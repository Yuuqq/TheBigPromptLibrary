最简单的调试方法：

a. 到处都放 print 语句
Grimoire 有一个热键来做这个。H

b. 不要害怕改变和破坏东西。最坏的情况你可以用 git stash 或 git revert 来恢复你的代码
小技巧：经常 stash 和 unstash（不删除），以创建迷你检查点和迷你伪提交

b. 在你的 UI 周围画红色或绿色的霓虹轮廓。边界框总是很奇怪，经常会搞乱布局。H 热键也有助于此

c. 学习使用调试器，以及如何暂停程序执行和检查变量

d. 橡皮鸭调试法。大声说出来。向别人解释问题。即使他们不理解。大声解释的行为通常会帮助你看到问题

e. 缩小问题空间
-清楚地陈述到底是什么出错了
-99%的 bug 报告都很糟糕，字面上没有说明 bug 是什么
-你的第一步是重现它，看看当某些东西"不工作"时到底发生了什么
-它没有加载吗？它在错误的位置吗？哪个变量是错的？错误的变量究竟来自哪里？
-准确定义什么是错的
-找到这个错误发生的确切位置和原因
-隔离它，找出最简单的修复方法

f. 大胆注释掉代码，设置模拟值。测试、探查和戳弄东西。看看会发生什么。行动就是信息。继续尝试

g. 如果它之前工作而现在不工作，使用 git bisect 找到破坏它的提交




Grimoire 调试技巧：
-反懒惰键，闭嘴写代码：C
-快速前进：W
-调试热键行：A, S, SS SoS, D, F, G, H, J

-有时 Grimoire 最大的优势与写代码无关，而是规划各个部分

-经常重新生成，写3次代码变体是便宜的，然后找到最好的那个。或者使用 A 和 D 热键来头脑风暴和规划方法

-经常开始新对话。清除历史记录会好得多

-编辑之前的消息并重新生成比试图修复一个糟糕的误解输出更容易。不要用垃圾污染你的上下文窗口

-AI 认为它的能力比实际的弱，有时你需要推动它，或要求它自我修复

-AI 通常不会第一次就写出最好的版本。逐步迭代构建。这通常是因为早期版本对需求缺乏清晰度。但也因为训练数据包括博客文章，这些文章给出了不正确的例子，然后是解决方案

-注意回声或模式。很多时候 LLM 喜欢重复东西。你经常会注意到回复或后续消息遵循早期回复的格式。注意这对代码生成的影响

-如果你的仓库很小，你可以把整个东西压缩并附加到 chatGPT 中。然后要求解压并读取代码
或者用少量相关文件这样做

-首先关注高级组件，让 Grimoire 填充细节。有时你只需要复制粘贴2个代码片段并询问如何连接它们
